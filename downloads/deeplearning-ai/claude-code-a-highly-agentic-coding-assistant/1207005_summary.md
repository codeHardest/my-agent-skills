## 详细内容（保留所有原始信息）

### 一、问题场景与调试方法论

视频开篇设置了一个真实场景：开发者时隔一段时间后重新使用应用程序，查询课程详情时出现错误。讲师资深的调试经验指出，很多人可能会直接复制错误信息给Claude AI，期望快速解决，但这并非最优方法。

视频倡导采用更系统化的方法：
- 不只是获取想要的答案或进行无目的的探索
- 利用已有的测试基础设施来程序化验证问题
- 在提示中不仅说明错误现象，还明确指定需要编写测试的位置
- 使用Claude Code的工具来系统地排查问题

这种方法的优势在于：通过测试先行，可以建立稳固的基础来进一步开发代码库，并且当出现问题时能够清楚地理解失败原因。

### 二、需要测试的关键文件

错误可能来源于多个Python文件的组合问题：

**AIGenerator相关**：
- 负责处理与Anthropic API的交互
- 可能存在prompt问题或获取响应的逻辑缺陷

**rag_system.py**：
- RAG（检索增强生成）系统的编排器
- 协调所有与Retrieval Augmented Generation相关的操作

**search_tools**：
- 定义底层工具
- 潜在的问题点之一

讲师的策略是让Claude为这些特定文件编写测试，然后运行测试来验证问题所在并进行相应修复。

### 三、测试实现过程

**启用Claude的高级能力**：
- 开启plan mode（计划模式）：在Claude开始编写测试前，先审批其理解的任务计划
- "Ask Claude to think a lot"：触发extended thinking，分配更多token用于思考过程
- 如有需要，可随时停止Claude的执行

**测试框架选择**：
- 使用pytest框架
- 对ChromaDB进行mock以获取必要的测试基础设施
- 创建单元测试和集成测试

**测试覆盖范围**：
- course_search_tool的测试
- AI generator的测试
- RAG system的测试
- vector store的测试

**依赖管理**：
- 使用UV工具安装依赖
- 确保pytest正常工作

### 四、Bug定位与修复

在编写测试的过程中，Claude识别出了一个关键的配置问题：MAX_RESULTS设置为0，这导致了向量搜索时返回的chunks数量为零。

**发现过程**：
- 测试运行结果提示max results存在问题
- 检查MAX_RESULTS配置发现值为0

**修复验证**：
- 修正MAX_RESULTS值后，测试按预期工作
- 验证能够正确获取期望的文件内容
- 在浏览器中新建对话，输入之前的相同问题
- 成功返回预期信息（课程详情），而不是之前的错误

这次修复不仅解决了当前的bug，还为项目建立了强大的测试基础设施，确保未来任何新修改都不会在不知情的情况下破坏现有功能。

### 五、代码重构：从单次工具调用到多轮调用

修复bug后，讲师提出了一个进一步的需求：重构ai_generator.py以支持多轮工具调用。

**当前限制**：
- ai_generator.py中设定每个查询最多一次搜索
- 这对简单查询是合理的
- 但对于复杂查询（如比较不同课程的提纲）需要多次工具调用

**重构目标**：
- 支持多次工具调用
- 能够迭代处理所有必要的工具
- 或通过递归方式解决问题

**代码现状分析**：
- ai_generator.py使用SDK直接与Anthropic模型通信
- 设置base_params并生成响应
- 构建系统prompt、处理消息和工具定义时没有迭代往返机制
- 缺少多轮对话和多次工具使用的积累逻辑

### 六、重构方案制定

**详细文档准备**：
- 创建backend-tool-refactor.md文件
- 在文档中详细描述当前行为和期望行为
- 提供示例流程：搜索与某课程讨论相同主题的另一课程
- 明确使用多个不同工具的场景
- 列出具体需求和注意事项

**并行方案生成策略**：
- 让Claude同时生成两个不同的重构方案
- 使用task工具并行调度两个子代理
- 每个子代理独立思考并提出方案
- 开发者从多个选项中选择最佳方案

**两个方案对比**：
- 方案A：较简单的迭代方法
- 方案B：更全面的多轮逻辑实现，带有不同的辅助方法
- 方案A更安全，方案B更适合长期维护

讲师选择了方案A，并启用plan mode获取详细计划后再执行。

### 七、方案A实施细节

**修改内容**：
1. 更新方法签名：增加最大轮次参数
2. 更新系统prompt
3. 在handle tool execution中实现核心重构逻辑

**预期行为**：
- 用户查询 → 多次工具调用 → 最终响应
- 保持向后兼容性
- 不修改内部RAG系统或API端点

**实现步骤**：
- 修改现有方法
- 更新测试用例
- 添加顺序工具调用的专门测试
- 运行测试验证实现正确性

**端到端验证**：

   - 询问课程lesson详情 → 获得课程信息和标题
   - 第一次工具调用返回课程提纲
   - 第二次工具调用返回具体lesson的详细信息和标题
3. 复杂查询测试：询问是否有其他课程涵盖与MCP课程Lesson 5相同的主题
   - 需要多次工具调用获取MCP课程信息和其他课程提纲
   - 进行重叠比较

本课展示了Claude Code在软件开发流程中的强大应用：
